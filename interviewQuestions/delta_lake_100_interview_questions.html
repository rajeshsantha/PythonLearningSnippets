📐 Fundamentals & Architecture Q1–Q10
1
What is Delta Lake and how does it differ from plain Parquet on ADLS/S3?
Easy
▼
✓ Strong Answer
Delta Lake is an open-source storage layer that adds ACID transactions, schema enforcement, scalable metadata handling, and time travel on top of Parquet files stored in object storage. Plain Parquet = immutable files, no atomicity, no versioning, no concurrent write safety. Delta adds a transaction log (_delta_log) — a WAL that tracks every operation (add/remove file, schema change, metadata). This gives you: (1) atomic commits, (2) snapshot isolation, (3) audit history, (4) schema enforcement on write.
✗ Weak Answer
"Delta Lake is like Parquet but better." — No specifics = no confidence. Must name the transaction log and ACID properties.
↳ Follow-ups

▸ "Can you use Delta Lake without Spark?" (Yes — Delta Standalone, delta-rs in Rust, Python deltalake library)

▸ "What does Delta Lake NOT give you that a traditional database does?" (No row-level locking, no primary key enforcement natively, no B-tree indexes)

Explain the Medallion Architecture (Bronze/Silver/Gold) and what Delta Lake features you use at each layer.
Easy
▼
✓ Strong Answer
Bronze: Raw ingestion. Schema-on-read. Enable CDF for downstream CDC. Append-only. Use autoLoader + cloudFiles for incremental ingestion. Retain source format.

Silver: Cleansed, validated, deduped. Schema enforcement on. Constraints for data quality. MERGE for upserts. Enable CDF to propagate changes to Gold.

Gold: Business-level aggregates. Optimized for BI. Run OPTIMIZE + ZORDER here. Partitioned by date/region. SLA-driven freshness.
↳ Follow-ups

▸ "When would you skip the Silver layer?" (Fast streaming paths with simple transforms, or when Bronze is already clean enough)

▸ "How do you handle a bug in Silver that corrupts 3 days of Gold data?" (Time travel to pre-bug Silver version, reprocess forward)

3
Delta Lake stores data as Parquet. So what exactly IS stored in the _delta_log directory?
Medium
▼
✓ Strong Answer
_delta_log contains:

1. JSON commit files (00000000000000000001.json): Each commit = one JSON file. Contains actions: add (new file path + stats), remove (deleted file path), metaData (schema, partitioning, table properties), protocol (min reader/writer version), commitInfo (who, when, what operation).

2. Checkpoint files (.parquet, every 10 commits): Aggregate all add/remove actions into a single Parquet file. Spark reads last checkpoint + JSON commits since → current snapshot. Avoids replaying thousands of JSON files.

3. _last_checkpoint file: JSON pointer to latest checkpoint version number.
↳ Follow-ups

▸ "What happens if a checkpoint file is corrupted?" (Delta falls back to replaying all JSON commits from version 0 — very slow on old tables)

▸ "How large can the _delta_log grow? What's the operational concern?" (1M commits = 1M JSON files = slow listing. Run VACUUM to clean old commit files, or enable log compaction)

What is the Delta Lake protocol version and why does it matter in production?
Easy
▼
✓ Strong Answer
Protocol action defines minReaderVersion and minWriterVersion — the minimum Delta Lake library versions that can read/write the table. When you enable features like Deletion Vectors, Liquid Clustering, or Column Mapping, Delta auto-upgrades the protocol. Risk: older Spark/Delta clients reading the table get an error: "Delta protocol version not supported." In production with mixed Delta versions across clusters, always check protocol compatibility before enabling new features with DESCRIBE DETAIL table_name.

Compare Delta Lake vs Apache Iceberg vs Apache Hudi. When would you choose each?
Medium
▼
✓ Strong Answer
Delta Lake: Best Spark/Databricks integration. Mature MERGE, CDF, Liquid Clustering. Strong on Azure (native Databricks). Weaker multi-engine support historically (improving with UniForm).

Apache Iceberg: Best multi-engine support (Spark, Flink, Trino, Presto, Snowflake). Hidden partitioning (no partition evolution downtime). Better for multi-cloud, multi-engine shops. AWS heavily invested.

Choose by: All-Databricks → Delta. Multi-engine / AWS → Iceberg. Sub-minute CDC upserts → Hudi.
⚠ Trap

Don't say "Delta is always better." Interviewers at non-Databricks shops (Snowflake, AWS, Meta) want to hear you know Iceberg/Hudi. Saying only Delta = tunnel vision.

6
What is the difference between Delta table partitioning and ZORDER? Are they the same thing?
Easy
▼
✓ Strong Answer
Partitioning: Hive-style directory partitioning. Creates subdirectories per partition value (e.g., /dt=2024-01-01/). Eliminates entire directories from scan via partition pruning. Best for low-cardinality filter columns (date, region, status).

ZORDER: Clustering within a partition (or whole table). Co-locates rows with similar column values in same Parquet files. Enables data skipping via min/max statistics per file. Best for high-cardinality filter columns (user_id, product_id) within a partition.

They are complementary, not alternatives. Common pattern: PARTITION BY date, ZORDER BY user_id within each date partition.
⚠ Classic Trap

ZORDER on a partitioned column = zero effect. Data already split by partition. Don't ZORDER by date if table is partitioned by date.

7
What are Deletion Vectors in Delta Lake and how do they improve DELETE/UPDATE performance?
Medium
▼
✓ Strong Answer
Traditional Delta DELETE rewrites entire Parquet files — expensive I/O for single-row deletes. Deletion Vectors (DVs) mark rows as deleted in a small bitmap file instead of rewriting the data file. The data file stays untouched; DVs are applied at read time to filter out deleted rows.

Benefits: Near-instant deletes (only write a small bitmap). No file rewrite = no write amplification. Files eventually compacted during OPTIMIZE.

Enable: ALTER TABLE t SET TBLPROPERTIES ('delta.enableDeletionVectors' = 'true')

Trade-off: Read overhead — must apply DV filter per file. DVs accumulate until OPTIMIZE compacts and physically removes deleted rows.

How does Delta Lake Column Mapping work and what problem does it solve?
Hard
▼
✓ Strong Answer
Without Column Mapping, renaming a column requires rewriting all Parquet files (the physical column name must match the schema name). Column Mapping decouples the logical column name (what SQL sees) from the physical column name stored in Parquet.

Enable: delta.columnMapping.mode = 'name'

Now you can: ALTER TABLE t RENAME COLUMN old_name TO new_name — zero file rewrite. The mapping is stored in the table metadata. Also enables dropping columns without rewriting files.

Required for: Live schema migration on production tables without downtime. Required by Databricks for UC (Unity Catalog) tables in some configurations.
↳ Follow-ups

▸ "What protocol version does Column Mapping require? Does this break older Delta readers?" (minReaderVersion=2, minWriterVersion=5 — yes, older Delta clients error out)

9
What is UniForm in Delta Lake 3.0+ and when would you use it?
Hard
▼
✓ Strong Answer
UniForm (Universal Format) writes both Delta AND Iceberg metadata for the same underlying Parquet files. Iceberg engines (Trino, Presto, Athena, Snowflake) can read the table as a native Iceberg table. Delta engines read it as Delta. No data duplication — same Parquet files, dual metadata.

Use case: Hybrid lakehouse where you write with Spark/Delta but need to query with non-Databricks engines (e.g., Athena for AWS teams, Snowflake for analytics). Eliminates "copy data out of Delta" anti-pattern.

Enable: TBLPROPERTIES ('delta.universalFormat.enabledFormats' = 'iceberg')

Limitation: Iceberg metadata generated async — slight lag before Iceberg readers see latest data. Not all Delta features map to Iceberg.
10
Explain Delta Lake's data skipping mechanism. What statistics does it collect and how are they used?
Medium
▼
✓ Strong Answer
On every write, Delta collects per-Parquet-file statistics and stores them in the transaction log's add action:

▸ numRecords: Row count per file
▸ minValues / maxValues: Per column, per file (first 32 columns by default — configurable via delta.dataSkippingNumIndexedCols)
▸ nullCount: Per column, per file

At query time with a WHERE clause, Delta reads only the stats from _delta_log (fast metadata read) and eliminates files where the filter is outside the min/max range. No Parquet files opened for eliminated files.

ZORDER maximizes this by co-locating data with similar values → narrower min/max ranges per file → more files skipped.
↳ Follow-ups

▸ "What happens to data skipping for string columns with high cardinality?" (Min/max on strings = lexicographic range — can be effective if data is sorted, poor if random)

▸ "Column 33 has no min/max stats. Why?" (Default only indexes first 32 columns. Change with ALTER TABLE SET TBLPROPERTIES('delta.dataSkippingNumIndexedCols'=50))

📋 Transaction Log Deep Dive Q11–Q18
11
How does Delta reconstruct the current table snapshot from the transaction log? Walk through the algorithm.
Medium
▼
✓ Strong Answer
1. Read _last_checkpoint → get latest checkpoint version N
2. Read checkpoint Parquet file at version N → get cumulative state (all active files)
3. List JSON commit files from version N+1 to latest (e.g., N+1.json, N+2.json, ...)
4. Apply each JSON's add/remove actions in order on top of checkpoint state
5. Result = current snapshot (set of active Parquet files)

Performance: Checkpoint every 10 commits means max 9 JSON files to read after checkpoint. Without checkpoints, must replay ALL commits from 0 (catastrophic on old tables).
↳ Follow-ups

▸ "What if _last_checkpoint is missing?" (Delta lists all JSON files in _delta_log, finds highest version, replays from 0 or finds latest checkpoint parquet by listing)

12
What does a typical Delta commit JSON file contain for an INSERT operation vs a DELETE operation?
Hard
▼
✓ Strong Answer
INSERT (append):
{"commitInfo": {"operation": "WRITE", "operationParameters": {"mode": "Append"}, "timestamp": ...}}
{"add": {"path": "part-00000.parquet", "size": 12345, "stats": "{\"numRecords\":1000, \"minValues\":{\"id\":1}, \"maxValues\":{\"id\":1000}}"}}

DELETE:
{"commitInfo": {"operation": "DELETE", ...}}
{"remove": {"path": "part-00000.parquet", "deletionTimestamp": 1234567890, "dataChange": true}}
{"add": {"path": "part-00001.parquet", "stats": "..."}} ← new file without deleted rows

DELETE = remove old file + add new file (rewrite). With DVs: remove old file's DV entry (if any), add new DV bitmap — no file rewrite.
13
Two Spark jobs try to commit to the same Delta table at the same time. How does Delta detect and resolve the conflict?
Hard
▼
✓ Strong Answer
Delta uses Optimistic Concurrency Control (OCC):

1. Writer A reads table at version 5, does work, tries to write version 6
2. Writer B also read version 5, tries to write version 6
3. Writer A commits first → version 6.json created atomically (object store put-if-absent)
4. Writer B tries to write version 6 → file already exists → conflict detected
5. Delta checks: did Writer A's commit affect the files Writer B is trying to write?
6. If NO overlap (e.g., different partitions) → Writer B re-reads, reapplies changes, commits as version 7
7. If YES overlap → ConcurrentModificationException thrown

Conflict types: ConcurrentAppendException, ConcurrentDeleteReadException, ConcurrentDeleteDeleteException, MetadataChangedException.
↳ Follow-ups

▸ "Two jobs APPEND to different partitions simultaneously. Does this conflict?" (No — append to non-overlapping partitions is conflict-free under Snapshot isolation)

▸ "Job A is reading partition dt=2024-01-01. Job B is deleting it. What happens?" (ConcurrentDeleteReadException if isolation = Serializable. Under Snapshot isolation, A continues with snapshot, B commits independently)

14
What is the difference between Serializable and WriteSerializable isolation levels in Delta Lake?
Hard
▼
✓ Strong Answer
WriteSerializable (default): Writes are serializable (no write conflicts). Reads may see stale data in some edge cases (concurrent write might not be visible immediately to a concurrent read). Best for most pipelines — higher concurrency, fewer conflicts.

Serializable: Both reads AND writes are serializable. Strongest isolation — matches traditional RDBMS. More conflicts, lower concurrency. Required for financial systems where read-your-own-write consistency matters.

Set: ALTER TABLE t SET TBLPROPERTIES ('delta.isolationLevel' = 'Serializable')
15
How often does Delta create checkpoint files and how would you tune this for a high-frequency write table?
Medium
▼
✓ Strong Answer
Default: checkpoint every 10 commits. Configurable: delta.checkpointInterval = 10 (table property).

For high-frequency writes (streaming, 100 commits/hour): reduce to 5 or even 1. Benefit: snapshot reconstruction faster (fewer JSON replays). Cost: more checkpoint Parquet writes.

For low-frequency large batch tables: keep at 10 or increase to 50. Checkpoint write overhead is significant for large tables.

Monitor: if snapshot read latency is high, reduce checkpoint interval. If write throughput is bottlenecked, increase it.
16
A Delta table has been running for 2 years with daily writes. The _delta_log directory has 730 JSON files. What problems can this cause and how do you fix them?
Medium
▼
✓ Strong Answer
730 JSON files is fine — checkpoints every 10 = max 9 JSONs to replay. The actual problem is the number of add/remove actions across all commits. Tables with millions of small files generate massive checkpoint Parquet files (billions of add entries).

Real problem at scale: Checkpoint Parquet becomes huge (GBs) → slow to read → slow snapshot construction.

Fixes: (1) OPTIMIZE regularly to compact small files (fewer add entries per checkpoint), (2) VACUUM to remove old remove entries, (3) Enable Delta log compaction (Databricks feature — compacts JSON commits into multi-part checkpoint), (4) Liquid Clustering reduces file churn vs ZORDER.
17
What is a multi-part checkpoint in Delta Lake and when is it used?
Hard
▼
✓ Strong Answer
When a table has millions of active files, a single checkpoint Parquet becomes enormous (slow to write and read). Multi-part checkpoints split the checkpoint into multiple Parquet files written in parallel.

Format: 00000000000000001000.checkpoint.0000000001.0000000003.parquet → checkpoint 1000, part 1 of 3.

The _last_checkpoint JSON records the number of parts. Reader reads all parts and merges.

Enabled by: delta.checkpoint.writeStatsAsStruct = true and configured in Databricks. Significantly improves performance on tables with 100M+ active files.

What is the metadata scalability problem in Delta Lake and how does Delta 3.x/Databricks address it?
Hard
▼
✓ Strong Answer
Problem: All file-level metadata (add/remove actions, stats) stored in _delta_log. For a table with 100M small files → checkpoint = 100M rows → multi-GB Parquet → slow to read just to plan a query.

Solutions:
▸ OPTIMIZE: Compacts files → fewer add entries needed
▸ Liquid Clustering: Manages file count automatically
▸ Databricks Photon + Delta metadata caching: Driver caches parsed checkpoint in memory across queries
▸ Delta 3.2+ Variant type + typed metadata: More efficient stats encoding
▸ Unity Catalog metadata accelerator: Externalizes metadata to a managed service (like a true catalog) bypassing _delta_log bottleneck for very large tables
🔒 ACID & Concurrency Q19–Q26
19
How does Delta Lake achieve atomicity? What happens if a Spark job crashes mid-write?
Medium
▼
✓ Strong Answer
Atomicity via commit-or-nothing model:

1. Spark writes Parquet data files to a temp _delta_log/.../_tmp path
2. Only when ALL files written successfully, a single JSON commit file is atomically written to _delta_log
3. Object stores (S3, ADLS) support atomic put-if-absent — guarantees only one writer wins per commit

If crash mid-write: Parquet files exist in storage but no commit JSON references them. Table readers never see these files (they don't appear in any snapshot). On next VACUUM run, orphaned files are cleaned up.

Orphan file cleanup: VACUUM table RETAIN 0 HOURS DRY RUN → shows orphaned files. Remove with VACUUM.
20
How do you make a Delta write idempotent? Why is this critical for Airflow retry scenarios?
Medium
▼
✓ Strong Answer
Idempotency strategies:

▸ overwrite + dynamic partition overwrite: Rewriting same partition multiple times = same result. Set spark.sql.sources.partitionOverwriteMode=dynamic. Safe for retry.

▸ MERGE with unique key: Same source data MERGEd twice = same result. MERGE is naturally idempotent on key.

▸ txnAppId + txnVersion: Delta's built-in idempotency. .option("txnAppId", "my_job").option("txnVersion", execution_date). Delta rejects duplicate (appId, version) pairs. Used in streaming for exactly-once.

Why critical for Airflow: Airflow retries re-run entire task. Without idempotency, retry = duplicate data. With dynamic overwrite or MERGE, retry = safe re-run.
21
Can two readers and one writer operate on a Delta table simultaneously without locking? Explain exactly what each sees.
Hard
▼
✓ Strong Answer
Yes — Delta uses MVCC (Multi-Version Concurrency Control) via snapshots. No read locks.

▸ Reader A starts at version 10 → reads snapshot 10 (fixed set of Parquet files)
▸ Writer commits version 11 (adds new files, marks old as removed)
▸ Reader A continues reading version 10 snapshot — unaffected by version 11
▸ Reader B starts after writer → reads version 11 (sees new data)

Reader A and Reader B see different data simultaneously — this is Snapshot Isolation. Neither reader blocks the writer. Writer doesn't block readers. No shared locks. This is fundamentally different from row-level locking in RDBMS.
22
Two MERGE operations run concurrently on the same Delta table targeting the same set of rows. What happens?
Hard
▼
✓ Strong Answer
Both MERGEs read the table at version N. One commits first (version N+1). Second tries to commit version N+1 — conflict detected.

Delta checks: did the first MERGE's commit touch any files that the second MERGE also read? Since both target the same rows = same files → ConcurrentModificationException.

Resolution options:
▸ Retry with backoff (Delta auto-retries up to a limit)
▸ Serialize MERGE operations (upstream queue per table partition)
▸ Partition table and assign different source batches to different partitions
▸ Use single streaming MERGE with foreachBatch (inherently serialized per batch)
23
What is the difference between overwrite mode and dynamic partition overwrite in Delta?
Medium
▼
✓ Strong Answer
Static overwrite (mode="overwrite"): Replaces the ENTIRE table (or partition specified via .partitionBy()). All partitions wiped and rewritten. Dangerous if table has multiple partitions and you only want to refresh one.

Dynamic partition overwrite: Only replaces partitions present in the source DataFrame. Other partitions untouched.

Enable: spark.conf.set("spark.sql.sources.partitionOverwriteMode", "dynamic") or per-write .option("partitionOverwriteMode", "dynamic")

Production pattern: Daily batch jobs use dynamic overwrite — refresh only today's partition without touching yesterday's. Safe for Airflow retries.
24
Your Delta table transaction log is corrupted at version 500. How do you recover?
Hard
▼
✓ Strong Answer
1. Identify extent: DESCRIBE HISTORY table — find last valid version
2. Delta FSCK (if available): FSCK REPAIR TABLE t — removes log entries referencing missing/corrupted files
3. If checkpoint exists before corruption: Delete corrupted JSON commits, latest checkpoint is valid. Table accessible up to last checkpoint.
4. Worst case — recreate from Parquet: CREATE TABLE t USING DELTA LOCATION 'path/to/parquet' — Delta scans existing Parquet files and creates a new _delta_log. Loses all history but table is accessible.
5. Prevention: Enable ADLS soft delete + versioning. Keep checkpoint interval low. Never run VACUUM with RETAIN 0 HOURS.
25
How do you roll back a Delta table to a previous version? Is it a true rollback?
Medium
▼
✓ Strong Answer
RESTORE command (Delta 0.8+):
RESTORE TABLE my_table TO VERSION AS OF 42
RESTORE TABLE my_table TO TIMESTAMP AS OF '2024-01-15 08:00:00'

Is it a true rollback? No — RESTORE creates a new commit that re-adds the files from the target version. The transaction log still contains all intermediate commits. History is preserved. This is RESTORE, not revert.

Important: RESTORE fails if VACUUM has removed the old version's files. Always test restore capability before vacuuming aggressively. Restore re-references old Parquet files, so those files must still exist in storage.
26
How do you enforce data quality constraints at the Delta table level (not application level)?
Hard
▼
✓ Strong Answer
Delta CHECK constraints:
ALTER TABLE t ADD CONSTRAINT positive_amount CHECK (amount > 0)
ALTER TABLE t ADD CONSTRAINT valid_status CHECK (status IN ('active', 'inactive', 'pending'))

Constraint evaluated on every write — any violating row fails the entire write (ACID). Listed in DESCRIBE DETAIL under properties.

NOT NULL constraints: ALTER TABLE t ALTER COLUMN id SET NOT NULL

Limitation: No UNIQUE or FOREIGN KEY constraints in Delta. For uniqueness, use MERGE + dedup logic or application-level checks + Great Expectations.
🔀 MERGE / UPSERT Q27–Q36
27
Write the SQL for a basic MERGE (upsert) on a Delta table — insert new, update existing by primary key.
Easy
▼ 
MERGE INTO target_table AS tgt
USING source_data AS src
ON tgt.id = src.id
WHEN MATCHED AND src.updated_at > tgt.updated_at THEN
  UPDATE SET
    tgt.name = src.name,
    tgt.amount = src.amount,
    tgt.updated_at = src.updated_at
WHEN NOT MATCHED THEN
  INSERT (id, name, amount, updated_at, created_at)
  VALUES (src.id, src.name, src.amount, src.updated_at, CURRENT_TIMESTAMP());
💡 Key Points
▸ Always include update condition (src.updated_at > tgt.updated_at) — prevents overwriting with stale data if source sends duplicates
▸ MERGE rewrites all matched files — expect increased write I/O
▸ Run OPTIMIZE after heavy MERGE batches to compact small files
28
Explain what Delta does internally when you run a MERGE. How many Spark jobs are triggered?
Medium
▼
✓ Strong Answer
MERGE internally executes:

Job 1 — Find matching files: Join source keys against Delta's file statistics (min/max) to identify which Parquet files contain matching rows. Data skipping applied here.

Job 2 — Apply changes: Read identified target files + source → apply MATCHED/NOT MATCHED logic → write new Parquet files. Non-matching target files untouched (copied by reference in commit JSON).

Commit: New JSON with add (new files) + remove (old rewritten files) actions.

Total: 2 Spark jobs + 1 metadata commit. For large targets with good data skipping, Job 1 eliminates most files → Job 2 is small.
↳ Follow-ups

▸ "If MERGE touches 90% of the table's files, what's the performance impact?" (Essentially a full table rewrite — consider INSERT OVERWRITE on full refresh instead)

Your MERGE is taking 2 hours. The target is 5TB partitioned by date. Source has 1M rows for only yesterday's date. How do you optimize?
Hard
▼
✓ Strong Answer
Add partition filter to MERGE condition:
ON tgt.id = src.id AND tgt.date = src.date

Delta now applies partition pruning — only reads/rewrites yesterday's partition. Without the date condition, Delta must scan all 5TB to find matching rows.

Additional optimizations:
▸ .option("spark.databricks.optimizer.mergeJoinRepartitionDeduplication", "true") — Databricks-specific, deduplicates source
▸ Pre-filter source to only changed records (don't send unchanged rows to MERGE)
▸ Ensure source is partitioned on same key as target before MERGE (avoid full shuffle)
▸ After MERGE: OPTIMIZE table WHERE date = yesterday ZORDER BY (id)

How do you delete rows in a Delta target table that are no longer present in the source using MERGE?
Hard
▼ 
-- WHEN NOT MATCHED BY SOURCE (Delta 2.4+ / Spark 3.4+)
MERGE INTO target AS tgt
USING source AS src
ON tgt.id = src.id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *
WHEN NOT MATCHED BY SOURCE THEN DELETE;
💡 Key Points
▸ "WHEN NOT MATCHED BY SOURCE" = rows in target with no matching source row → DELETE
▸ Use with caution — if source is a partial update (not full snapshot), this deletes valid rows. Only use when source is a complete replacement of the target dataset.
▸ Add partition filter: WHEN NOT MATCHED BY SOURCE AND tgt.date = '2024-01-15' THEN DELETE to scope deletes.
31
Design a MERGE-based SCD Type 2 pattern for Delta. How do you handle the "close old record, open new record" requirement in a single MERGE?
Hard
▼
✓ Strong Answer
Single MERGE can't easily insert a new row while updating an existing one. Standard pattern = 2-pass approach:

Pass 1: MERGE to close current record (UPDATE end_date, set is_current=false)
Pass 2: INSERT new current records (for changed rows + brand new rows)

Alternatively, UNION trick in single MERGE: Source = changed rows UNION same rows with a "close" flag. Match on business key + is_current — one WHEN clause updates/closes, another inserts new.
-- Union approach: stage records to close + new records
WITH staged AS (
  -- rows to close: matched + actually changed
  SELECT src.*, 'UPDATE' AS _action FROM source src
  JOIN target tgt ON src.id = tgt.id AND tgt.is_current = true
  UNION ALL
  -- rows to insert as new current
  SELECT src.*, 'INSERT' AS _action FROM source src
)
MERGE INTO target tgt USING staged src
ON tgt.id = src.id AND tgt.is_current = true AND src._action = 'UPDATE'
WHEN MATCHED THEN UPDATE SET is_current = false, end_date = src.updated_at
WHEN NOT MATCHED AND src._action = 'INSERT' THEN INSERT *;
32
Source DataFrame has a new column not present in the target Delta table. What happens during MERGE and how do you handle it?
Hard
▼
✓ Strong Answer
Default: MERGE fails with AnalysisException — "cannot resolve column new_col in target schema."

Fix — enable schema evolution in MERGE:
spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true")
or per-operation: DeltaTable.forPath(...).merge(...).option("mergeSchema", "true")

With autoMerge: new column added to target schema, existing rows get NULL for new column.

Risk: Accidental schema drift from malformed source. Best practice: validate source schema against expected schema before MERGE, only enable autoMerge in bronze layer.
33
Source DataFrame for MERGE has duplicate rows for the same primary key. What happens and how do you fix it?
Hard
▼
✓ Strong Answer
MERGE with duplicate source keys = non-deterministic behavior. For the same target row matched by multiple source rows, which UPDATE wins is undefined. Delta may raise an error or apply one non-deterministically.

Fix — dedup source before MERGE:
# Deduplicate source: keep latest by timestamp per key
from pyspark.sql import Window
w = Window.partitionBy("id").orderBy(F.col("updated_at").desc())
source_deduped = source_df \
    .withColumn("_rn", F.row_number().over(w)) \
    .filter("_rn = 1") \
    .drop("_rn")
# Now safe to MERGE

How do you write a MERGE that updates only specific columns (not all) and only when the value has actually changed?
Hard
▼ 
MERGE INTO target tgt
USING source src ON tgt.id = src.id
WHEN MATCHED AND (
    tgt.status != src.status OR
    tgt.amount != src.amount OR
    tgt.name != src.name
) THEN
  UPDATE SET
    tgt.status = src.status,
    tgt.amount = src.amount,
    tgt.name = src.name,
    tgt.updated_at = CURRENT_TIMESTAMP()
    -- Note: id, created_at NOT updated
WHEN NOT MATCHED THEN INSERT *;
💡 Key Points
▸ The WHEN MATCHED condition prevents no-op updates — files only rewritten when data actually changed. Reduces write amplification.
▸ Watch for NULL comparison: tgt.col != src.col is false when either is NULL. Use NOT (tgt.col <=> src.col) for NULL-safe comparison in Spark SQL.
35
How do you use MERGE inside a Spark Structured Streaming foreachBatch to achieve exactly-once upserts?
Hard
▼ 
def upsert_to_delta(batch_df, batch_id):
    from delta.tables import DeltaTable
    
    # Dedup within batch first (Kafka may redeliver)
    deduped = batch_df.dropDuplicates(["id"])
    
    if deduped.isEmpty(): return
    
    tgt = DeltaTable.forPath(spark, "/mnt/delta/target")
    tgt.alias("t").merge(
        deduped.alias("s"),
        "t.id = s.id"
    ).whenMatchedUpdateAll() \
     .whenNotMatchedInsertAll() \
     .execute()

query = stream_df.writeStream \
    .foreachBatch(upsert_to_delta) \
    .option("checkpointLocation", "/mnt/ckpt/target") \
    .start()
💡 Why Exactly-Once
Checkpoint tracks Kafka offsets. On restart, same batch_id is replayed. MERGE is idempotent — same source data MERGEd again = same result. Dedup within batch handles Kafka at-least-once delivery. Delta commit = atomic. Combined: exactly-once semantics.

What is "write amplification" in the context of Delta MERGE and how do you minimize it?
Hard
▼
✓ Strong Answer
Write amplification = amount of data written exceeds the actual changed data. MERGE must rewrite entire Parquet files even if only 1 row in each file changed.

Example: 1000 files × 100MB each = 100GB table. Source has 1 row in each file → all 1000 files rewritten → 100GB write for 1KB of changes.

Minimize:
▸ Partition on write: Changes concentrated in today's partition → only rewrite today's files
▸ ZORDER source column: Changes clustered in fewer files → fewer files rewritten
▸ Deletion Vectors: No file rewrite for deletes/updates — just a bitmap
▸ Low-shuffle MERGE: Add partition key to MERGE condition for partition pruning
▸ Larger files: Fewer total files → fewer rewrite operations
⚡ OPTIMIZE, ZORDER & Liquid Clustering Q37–Q46
37
What does OPTIMIZE do in Delta Lake? What is the target file size?
Easy
▼
✓ Strong Answer
OPTIMIZE compacts small Parquet files into larger ones using a bin-packing algorithm. Target file size: 1GB by default (configurable: delta.targetFileSize). Files already at target size untouched.

What it does: Reads small files within a partition, writes fewer large files, removes old small files via _delta_log (add new + remove old).

What it does NOT do: Does NOT globally sort data. Does NOT remove old versions (that's VACUUM). Does NOT reduce data — same rows, fewer files.

When to run: After streaming writes, after many MERGEs, after initial data loads that created many small files. Schedule: daily or after threshold of small files detected.
38
Explain how Z-Order curves work mathematically and why they improve multi-dimensional data locality.
Medium
▼
✓ Strong Answer
Z-Order (Morton code) interleaves the bits of multiple column values into a single integer. Example with (user_id=5=101, country=3=011): interleaved bits = 011011. Rows sorted by this interleaved value are co-located in Parquet files.

Why it works: Nearby points in 2D space (user_id, country) remain nearby in the 1D Z-curve sort order — much better locality than sorting on one column at a time (which would cluster by first column only).

Effect on data skipping: Files contain data that's similar on ALL Z-ordered columns simultaneously. A query like WHERE user_id BETWEEN 100 AND 200 AND country = 'US' can skip many files because their min/max stats won't overlap with the query range.

Limitation: Z-order effectiveness degrades with 4+ columns (curse of dimensionality — the Z-curve fills the space less efficiently).

What is the difference between OPTIMIZE ZORDER BY and a regular SORT BY in Spark before writing? When would you choose each?
Hard
▼
✓ Strong Answer
SORT BY (single column): Globally sorts by one column. Min/max range per file is perfectly tight for that column → maximum data skipping for that column. BUT adjacent files may have overlapping ranges for other columns.

ZORDER BY (multi-column): Interleaves multiple columns. No column is perfectly sorted, but ALL specified columns have tighter-than-random ranges per file. Better for multi-column filters.

Choose SORT: Queries always filter on exactly one high-cardinality column. Simple and predictable.

Choose ZORDER: Queries filter on 2-4 columns in combination. Common in analytics (event_date + user_id, country + product_id).

OPTIMIZE ZORDER vs df.sort().write: OPTIMIZE is incremental (skips already-sorted files), post-hoc, and triggers via Delta. df.sort() on write is one-time, requires full shuffle on every write. For ongoing pipelines: OPTIMIZE ZORDER scheduled periodically wins.

What is Liquid Clustering and how does it solve the limitations of ZORDER? What are its trade-offs?
Hard
▼
✓ Strong Answer
ZORDER limitations: Not incremental — rewrites ALL files even if 99% already clustered. Expensive for large tables. Cluster keys baked into OPTIMIZE command (not table-level). Can't change Z-order columns without full rewrite.

Liquid Clustering solves all of these:
▸ Incremental: Only clusters files that need it. Amortizes cost over time.
▸ Table-level definition: CLUSTER BY (user_id, event_date) at CREATE TABLE
▸ Evolve cluster keys: ALTER TABLE t CLUSTER BY (new_col) — future OPTIMIZE uses new key
▸ No partition required: Works without Hive partitioning (fewer small directory problems)

Trade-offs: Still requires OPTIMIZE to run (just cheaper). Not yet available in all Delta versions (3.1+). May produce slightly different data locality than hand-tuned ZORDER for very specific query patterns.

What are Bloom Filter indexes in Delta Lake? When would you use them over ZORDER?
Hard
▼
✓ Strong Answer
Bloom Filters are probabilistic data structures that answer "is this value definitely NOT in this file?" with no false negatives (false positives possible — sometimes reads unnecessary files).

Create: CREATE BLOOMFILTER INDEX ON TABLE t FOR COLUMNS (id OPTIONS (fpp=0.1, numItems=50000000))

Use over ZORDER when:
▸ Queries are point lookups (WHERE id = 'abc123') — ZORDER helps range scans, Bloom filters help point lookups
▸ Column has high cardinality with random distribution (UUIDs, hash IDs) — ZORDER can't cluster random data
▸ Data is already physically clustered (you don't need rewrite, just add an index)

What is "Auto Optimize" in Databricks and how does it interact with OPTIMIZE ZORDER?
Hard
▼
✓ Strong Answer

What is the relationship between VACUUM and Time Travel? What is the default retention period and how do you change it?
Medium
▼
✓ Strong Answer

What is the difference between a Delta Shallow Clone and Deep Clone? When would you use each?
Hard
▼
✓ Strong Answer

CREATE TABLE backup_table DEEP CLONE prod_table LOCATION '/mnt/backup/prod'

📊 Schema Evolution & Enforcement Q55–Q62
55
What is the difference between schema enforcement and schema evolution in Delta Lake?
Easy
▼
✓ Strong Answer
Schema Enforcement (on by default): Rejects writes that don't match the existing table schema. New columns in source → AnalysisException. Columns missing in source → NULL for missing (configurable). Protects data quality by preventing unexpected schema drift.

Schema Evolution (opt-in): Allows schema to automatically expand when new columns arrive in source. Enable: .option("mergeSchema", "true") per write, or spark.conf.set("spark.databricks.delta.schema.autoMerge.enabled", "true") globally.

What schema evolution does NOT handle: Column type changes (int → string), column drops, column renames — these require ALTER TABLE or column mapping.

A source system changes a column from INTEGER to BIGINT. How do you handle this in Delta Lake?
Medium
▼
✓ Strong Answer
INT → BIGINT is a widening type change — safe (no data loss). Delta supports this via mergeSchema.

Delta widens the column type in the schema. Old Parquet files with INT are read as BIGINT automatically (Parquet reader handles widening).

Narrowing type change (BIGINT → INT): Rejected — data loss risk. Must use overwriteSchema=true (full table rewrite) or cast explicitly before writing.

ALTER TABLE approach: ALTER TABLE t CHANGE COLUMN amount amount BIGINT — updates schema metadata, no file rewrite needed for widening.

Design a CDC pipeline using Delta CDF that propagates Silver table changes to a Gold aggregation table incrementally (not full recompute).
Hard
▼
✓ Strong Answer
Pattern: read Silver CDF → compute delta aggregates → MERGE into Gold
def incremental_gold_update(batch_df, batch_id):
    # batch_df = CDF changes from Silver since last checkpoint
    
    # Compute revenue delta from inserts/updates
    inserts_posts = batch_df.filter(
        F.col("_change_type").isin(["insert", "update_postimage"])
    )
    # Subtract preimage revenue (reversal of old value)
    updates_pres = batch_df.filter(
        F.col("_change_type") == "update_preimage"
    ).withColumn("amount", F.col("amount") * -1)  # negate
    deletes = batch_df.filter(
        F.col("_change_type") == "delete"
    ).withColumn("amount", F.col("amount") * -1)
    
    delta_agg = inserts_posts.union(updates_pres).union(deletes) \
        .groupBy("product_id", "date") \
        .agg(F.sum("amount").alias("revenue_delta"))
    
    # MERGE delta into Gold (additive)
    DeltaTable.forPath(spark, "/mnt/gold/revenue") \
        .alias("g").merge(delta_agg.alias("d"), "g.product_id=d.product_id AND g.date=d.date") \
        .whenMatchedUpdate(set={"revenue": F.col("g.revenue") + F.col("d.revenue_delta")}) \
        .whenNotMatchedInsert(values={"product_id": F.col("d.product_id"), "date": F.col("d.date"), "revenue": F.col("d.revenue_delta")}) \
        .execute()

spark.readStream.format("delta") \
    .option("readChangeFeed", "true") \
    .option("startingVersion", 1) \
    .table("silver.orders") \
    .writeStream.foreachBatch(incremental_gold_update) \
    .option("checkpointLocation", "/mnt/ckpt/gold_revenue") \
    .start()

Does CDF capture changes made by INSERT OVERWRITE or REPLACE TABLE? What about OPTIMIZE?
Hard
▼
✓ Strong Answer
INSERT OVERWRITE: CDF does NOT capture individual row changes for overwrites. The operation is recorded in history but no pre/post image per row. CDF consumers reading this version see all rows as inserts.

REPLACE TABLE: Treated as DDL operation. CDF returns error if consumer tries to read across this boundary. Consumer must do full resync.

OPTIMIZE: Does NOT generate CDF changes. OPTIMIZE rewrites files but doesn't logically change data. CDF consumers skip OPTIMIZE commits (no _change_data written).

RESTORE: Generates CDF changes — rows that were deleted but restored appear as inserts. Rows that were present but removed appear as deletes.

Implication: CDF works best with MERGE, DELETE, UPDATE operations. Avoid INSERT OVERWRITE on CDF-enabled tables if downstream consumers need row-level changes.

▸ WHEN MATCHED THEN UPDATE: Produces update_preimage (old row) + update_postimage (new row)
▸ WHEN NOT MATCHED THEN INSERT: Produces insert (new row only, no preimage)
▸ WHEN MATCHED THEN DELETE: Produces delete (old row, no postimage)
▸ WHEN NOT MATCHED BY SOURCE THEN DELETE: Produces delete for each target row without a source match

How does Spark Structured Streaming read from a Delta table as a streaming source? What does it track and how?
Medium
▼
✓ Strong Answer
Delta source uses version-based progress tracking instead of Kafka-style offsets. Checkpoint stores the last Delta version processed. On each trigger:

1. Read _delta_log to find new commits since last checkpoint version
2. Process only new add actions (new files) from those commits
3. Update checkpoint with new max version

How does Delta Lake achieve exactly-once when used as a streaming sink? What happens on job restart?
Hard
▼
✓ Strong Answer
Delta streaming sink uses epoch-based idempotency:

Each micro-batch has a unique epoch ID (batch_id). Delta stores this in the transaction log commit as txnAppId + txnVersion.

On job restart: Spark reads last successful epoch from checkpoint. Replays that epoch. Delta checks: "Did I already commit epoch N for this appId?" If yes → commit rejected (duplicate detected). Spark advances to epoch N+1.

On restart behavior: Source re-reads data from checkpoint offset. Delta sink rejects already-committed epochs. Net result: no duplicates in Delta even with Kafka at-least-once source.

Requirement: Must use consistent checkpointLocation across restarts. Changing checkpoint = lose epoch tracking = potential duplicates.

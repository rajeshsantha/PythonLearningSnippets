D1

Spark Internals & Performance Tuning

Execution model · DAG · Shuffle · Memory · AQE · Skew · Partitioning

🧠 Concepts to Learn (Deep) ▼

1. Execution Model

  • Driver vs Executor: Driver orchestrates, builds DAG, schedules tasks. Executor runs tasks in JVM processes. Know memory layout of each.
  • DAG → Stages → Tasks: Spark converts transformations into a DAG. Stages separated by shuffle boundaries. One task per partition per stage.
  • Lazy Evaluation: Nothing executes until an action (.count(), .write(), .collect()). Understand how Catalyst optimizer rewrites the logical plan into physical plan.
  • Narrow vs Wide Transformations: map/filter are narrow (no shuffle). groupBy/join/distinct are wide (trigger shuffle). Wide = new stage.
  • Catalyst Optimizer: Logical Plan → Analyzed Logical Plan → Optimized Logical Plan → Physical Plans → Best Physical Plan (cost model). Know predicate pushdown, column pruning, constant folding.
  • Tungsten Engine: Off-heap binary encoding. Whole-stage code generation (codegen). Avoids JVM object overhead. Know UnsafeRow format.

2. Memory Management

  • Unified Memory Model: spark.executor.memory → Storage (cache) + Execution (shuffle, join) regions. Dynamic sharing (one borrows from other).
  • OOM Causes: Large shuffles, too many cached RDDs, collect() on huge dataset, bad UDFs creating large objects.
  • spark.memory.fraction (0.6) + spark.memory.storageFraction (0.5): Know what happens when storage memory is evicted under execution pressure.
  • Off-heap memory: spark.memory.offHeap.enabled, useful for large caches bypassing GC.

3. Shuffle Deep Dive

  • Sort-Merge Shuffle (default): Map side writes sorted files per reducer. Reduce side reads and merges. Disk spill behavior.
  • spark.sql.shuffle.partitions (default 200): Critical to tune. Too few = OOM/skew, too many = task overhead. Rule of thumb: 2-3x cores, or target 128–512 MB per partition.
  • Tungsten Sort Shuffle (bypass): For small datasets / narrow ops — no sort, direct write. Triggers when spark.shuffle.manager=tungsten-sort and partitions ≤ 200.

4. Adaptive Query Execution (AQE)

  • What it solves: Re-optimizes query mid-flight using runtime statistics. Enabled by spark.sql.adaptive.enabled=true (default in Spark 3.2+).
  • 3 Features: (a) Dynamic coalescing of shuffle partitions. (b) Dynamic switching of join strategies (SortMerge → BroadcastHash). (c) Skew join optimization (splits skewed partitions).
  • Limitations: Works only on Spark SQL queries (not RDD API). Overhead of collecting stats between stages.

5. Data Skew

  • Salting technique: Add random salt (0-N) to skewed key, explode and join with replicated small table. Remove salt post-join.
  • Broadcast Join: For tables < spark.sql.autoBroadcastJoinThreshold (10MB default). spark.broadcast.blockSize tuning.
  • AQE Skew Join: Automatically splits skewed partitions when spark.sql.adaptive.skewJoin.enabled=true. Know thresholds: skewedPartitionFactor and skewedPartitionThresholdInBytes.

6. Partitioning Strategies

  • repartition() vs coalesce(): repartition = full shuffle, can increase/decrease. coalesce = no shuffle, can only decrease (may create uneven partitions).
  • partitionBy() on write: Hive-style partitioning on disk. Choose low-cardinality columns. Avoid high-cardinality = many small files.
  • bucketBy(): Pre-shuffles data at write time into fixed buckets. Eliminates shuffle on reads for same-bucketed joins. Requires persistent table (not path-based).

PySpark — Salting to Fix Skew

pyspark · production-level · skew join with salting
from pyspark.sql import SparkSession
from pyspark.sql import functions as F
from pyspark.sql.types import IntegerType

spark = SparkSession.builder \
    .config("spark.sql.adaptive.enabled", "true") \
    .config("spark.sql.adaptive.skewJoin.enabled", "true") \
    .getOrCreate()

# Large fact table with skewed user_id (user_id=0 = 80% of rows)
fact_df = spark.table("events.user_events")  # 500GB
dim_df  = spark.table("dim.users")             # 5GB

SALT_FACTOR = 50  # tune based on skew severity

# Step 1: Salt the fact table
fact_salted = fact_df.withColumn(
    "salt", (F.rand() * SALT_FACTOR).cast(IntegerType())
).withColumn(
    "salted_user_id", F.concat_ws("_", F.col("user_id"), F.col("salt"))
)

# Step 2: Explode the dimension table with all salt values
salt_range = spark.range(SALT_FACTOR).select(
    F.col("id").cast(IntegerType()).alias("salt")
)
dim_exploded = dim_df.crossJoin(salt_range).withColumn(
    "salted_user_id", F.concat_ws("_", F.col("user_id"), F.col("salt"))
).drop("salt")

# Step 3: Join on salted key
result = fact_salted.join(
    F.broadcast(dim_exploded) if dim_df.count() * SALT_FACTOR < 10_000_000 else dim_exploded,
    on="salted_user_id",
    how="inner"
).drop("salt", "salted_user_id")

result.write.format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .save("/mnt/output/events_enriched")

# Expected: partition task time std deviation drops from 40min to ~2min

SQL — Window Function Challenge

sql · advanced · rolling 7-day revenue per user segment
-- Find users whose 7-day rolling spend increased >20% vs prior 7-day period
-- and whose current week rank in their segment improved
WITH daily_spend AS (
    SELECT
        user_id, segment,
        event_date,
        SUM(amount) AS daily_total
    FROM transactions
    WHERE event_date >= DATEADD(day, -21, CURRENT_DATE())
    GROUP BY user_id, segment, event_date
),
rolling_window AS (
    SELECT
        user_id, segment, event_date,
        SUM(daily_total) OVER (
            PARTITION BY user_id
            ORDER BY event_date
            ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
        ) AS rolling_7d_curr,
        SUM(daily_total) OVER (
            PARTITION BY user_id
            ORDER BY event_date
            ROWS BETWEEN 13 PRECEDING AND 7 PRECEDING
        ) AS rolling_7d_prev
    FROM daily_spend
),
ranked AS (
    SELECT
        user_id, segment, event_date,
        rolling_7d_curr, rolling_7d_prev,
        DENSE_RANK() OVER (
            PARTITION BY segment, event_date
            ORDER BY rolling_7d_curr DESC
        ) AS curr_rank,
        LAG(DENSE_RANK() OVER (
            PARTITION BY segment, event_date
            ORDER BY rolling_7d_curr DESC
        ), 7) OVER (
            PARTITION BY user_id
            ORDER BY event_date
        ) AS prev_rank
    FROM rolling_window
    WHERE event_date = CURRENT_DATE() - 1
)
SELECT *
FROM ranked
WHERE
    rolling_7d_prev > 0
    AND rolling_7d_curr / rolling_7d_prev > 1.20
    AND curr_rank < prev_rank  -- rank improved (lower = better)
ORDER BY segment, curr_rank;

⚠️ Trick Questions & Common Mistakes ▼
Trick 1: "Does cache() guarantee data stays in memory?"

No. If memory is needed for execution, storage region can be evicted (LRU). cache() is a hint, not a guarantee. persist(DISK_AND_MEMORY) spills to disk.

Trick 2: "You increase shuffle partitions to 2000. Job gets slower. Why?"

Too many partitions = task scheduling overhead > parallelism benefit. Each task has 5-10ms overhead. 2000 tasks × 10ms = 20 seconds just in scheduling per stage.

Trick 3: "Why does coalesce() produce uneven partitions?"

coalesce combines adjacent partitions (no shuffle). If original partitions have unequal data, merged partitions inherit that imbalance. repartition reshuffles evenly.

Common Mistake: Confusing partitionBy (write) with repartition (in-memory)

partitionBy on write creates Hive-style directory partitions. repartition creates in-memory task partitions. They serve completely different purposes. Mixing them up in interviews = red flag.

Delta Lake — Deep Internals

ACID · Transaction Log · MERGE · OPTIMIZE · ZORDER · Time Travel · Vacuum · CDC

🧠 Concepts to Learn (Deep) ▼

1. Delta Transaction Log (_delta_log)

  • JSON commit files: Every write/update/delete creates a new JSON file (00000000000000000001.json). Contains: add/remove file actions, commitInfo, metadata, protocol.
  • Checkpoint files (.parquet): Created every 10 commits. Aggregates all JSON actions. Speeds up snapshot reconstruction (no need to replay all JSONs).
  • Snapshot = last checkpoint + JSON commits since checkpoint. Delta reads these to determine current state of the table.
  • Optimistic Concurrency Control: Writers read current version, make changes, then try to commit. If another writer committed in between, conflict resolution kicks in (can concurrent deletes on non-overlapping partitions succeed? Yes.)

2. MERGE (UPSERT) — The Most Interviewed Topic

  • Performance internals: Delta MERGE does a join between source and target, then generates INSERT/UPDATE/DELETE actions. Rewrites entire affected files.
  • Optimization: Use partition pruning in MERGE conditions. Low shuffle with partitioned target. Small file problem after many MERGEs → run OPTIMIZE.
  • WHEN NOT MATCHED BY SOURCE: Deletes rows in target not present in source (Spark 3.4+/Delta 2.4+). Critical for CDC pipelines.
  • Schema evolution in MERGE: .option("mergeSchema", "true") for new columns in source. NOT automatic by default.

3. OPTIMIZE + ZORDER

  • OPTIMIZE: Compacts small files into larger ones (target: 1GB by default). Uses bin-packing algorithm. Does NOT sort data globally.
  • ZORDER: Co-locates related data using Z-order space-filling curve. Multidimensional clustering. Best on 1-4 high-cardinality filter columns.
  • ZORDER limitation: Rewrites all files. Expensive on large tables. Not incremental. Only benefits queries that filter on Z-ordered columns.
  • Liquid Clustering (Delta 3.1+): Replaces ZORDER. Incremental, on-write clustering. Cluster keys defined at table level. You MUST know this for 2024/2025 interviews.
  • Data Skipping: Delta stores min/max stats per column per file. ZORDER maximizes data skipping effectiveness.

4. Time Travel

  • VERSION AS OF / TIMESTAMP AS OF reads snapshot at that version.
  • Use case: Debugging, rollback, audit trail, reproducible ML features.
  • Limit: VACUUM removes old files (default 7 days retention). After vacuum, old versions inaccessible.
  • Rollback pattern: INSERT OVERWRITE from historical version — NOT a true rollback of transaction log.

5. Change Data Feed (CDF)

  • Enable: delta.enableChangeDataFeed = true on table.
  • Read changes: spark.readStream.option("readChangeData", true).table("my_table")
  • Columns added: _change_type (insert/update_preimage/update_postimage/delete), _commit_version, _commit_timestamp.
  • Use case: Streaming CDC pipelines, incremental aggregations, downstream event-driven systems.

Production-Level MERGE with SCD Type 2

pyspark · delta lake · SCD Type 2 MERGE pattern
from delta.tables import DeltaTable
from pyspark.sql import functions as F
from datetime import datetime

def upsert_scd2(spark, source_df, target_table_path: str, key_cols: list, ts_col: str):
    """
    SCD Type 2 MERGE: closes old record, inserts new record.
    source_df: incoming changes with key + all attributes + ts_col
    """
    delta_tbl = DeltaTable.forPath(spark, target_table_path)
    
    # Tag source rows
    source_tagged = source_df.withColumn("_merge_key", 
        F.concat_ws("||", *[F.col(c) for c in key_cols])
    )
    
    # Stage 1: MERGE to close old records (UPDATE end_date)
    delta_tbl.alias("tgt").merge(
        source_tagged.alias("src"),
        " AND ".join([f"tgt.{c} = src.{c}" for c in key_cols]) +
        " AND tgt.is_current = true"
    ).whenMatchedUpdate(
        condition="src." + ts_col + " > tgt." + ts_col,
        set={
            "is_current": F.lit(False),
            "end_date": F.col("src." + ts_col),
            "_updated_at": F.current_timestamp()
        }
    ).execute()
    
    # Stage 2: INSERT new records for changed rows + brand new rows
    new_rows = source_tagged.join(
        delta_tbl.toDF().filter("is_current = true"),
        on=key_cols, how="left_anti"
    ).union(
        source_tagged.join(
            delta_tbl.toDF().filter("is_current = false"),
            on=key_cols, how="inner"
        ).select(source_tagged.columns)
    ).withColumn("is_current", F.lit(True)) \
     .withColumn("start_date", F.col(ts_col)) \
     .withColumn("end_date", F.lit(None).cast("timestamp"))
    
    new_rows.write.format("delta") \
        .mode("append") \
        .option("mergeSchema", "true") \
        .save(target_table_path)

# OPTIMIZE + ZORDER after heavy MERGE batch
def post_merge_maintenance(spark, table_path: str, zorder_cols: list):
    spark.sql(f"""
        OPTIMIZE delta.`{table_path}`
        ZORDER BY ({', '.join(zorder_cols)})
    """)
    spark.sql(f"VACUUM delta.`{table_path}` RETAIN 168 HOURS")  # 7 days
⚠️ Trick Questions & Common Mistakes ▼
Trick: "Can you VACUUM a table and still do Time Travel to yesterday?"

Only if VACUUM retains > 24 hours AND no VACUUM has run with shorter retention. The default is 7 days (168 hours). NEVER run VACUUM with RETAIN 0 HOURS in production — you'll lose all history.

Trick: "Is OPTIMIZE idempotent?"

Yes, but it creates new commit versions each time it runs (even if nothing to compact). This bloats the transaction log. Run OPTIMIZE on a schedule, not after every write.

Common Mistake: Using ZORDER on partition columns

ZORDER on a partitioned column (e.g., date if table is partitioned by date) has zero effect — data is already split by partition. ZORDER within a partition. Classic interview trap.

PySpark Coding + Advanced SQL

UDFs · Window Functions · SCD · CDC · Joins · Subqueries · Performance Anti-Patterns

🧠 Concepts to Learn (Deep) ▼

1. UDF Performance Deep Dive

  • Python UDF cost: Each row serialized from JVM → Python process (Py4J), executed in Python, deserialized back. Can be 100x slower than native Spark operations.
  • Pandas UDF (Vectorized): Data sent as Arrow batches. 10-100x faster than row-wise Python UDFs. Use @pandas_udf decorator. Iterator pattern for large state.
  • Scala UDF: Runs in JVM — same performance as built-in functions. Best choice for complex logic in performance-critical paths.
  • When to use which: Built-in → Pandas UDF → Scala UDF → Python UDF (last resort).

2. Window Function Mastery

  • Frame types: ROWS BETWEEN (physical offset) vs RANGE BETWEEN (value-based offset). Know the difference — RANGE can include duplicates.
  • LAG/LEAD: Offset with optional default. Use for gap analysis, session detection, change detection.
  • FIRST_VALUE / LAST_VALUE: Frame defaults are different! LAST_VALUE needs explicit ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW or you'll get wrong results.
  • Session windows: Detect sessions using LAG + CASE to flag session breaks, then cumulative SUM to assign session IDs.

3. SCD Implementations in SQL

  • SCD Type 1: Overwrite. Simple UPDATE. Loses history. Use for non-critical attributes.
  • SCD Type 2: Insert new row, close old with end_date + is_current flag. Maintains full history. Use for audit, point-in-time reporting.
  • SCD Type 3: Add prev_value column. Limited history (only one prior value). Rare in practice.
  • SCD Type 6 (hybrid 1+2+3): SCD2 + current_value denormalized on all rows. Eliminates self-join for current value lookup.

4. CDC Patterns

  • Log-based CDC (Debezium): Reads DB transaction log. Captures inserts/updates/deletes with before/after images. Near-zero impact on source.
  • Query-based CDC: Uses updated_at watermark. Misses hard deletes (use soft-delete + is_deleted flag). Simple but limited.
  • Processing CDC events: Apply in order (sort by LSN or timestamp). Handle out-of-order events. Dedup by primary key + latest timestamp.

PySpark — Session Detection + Funnel Analysis

pyspark · window functions · session detection
from pyspark.sql import functions as F, Window

# Schema: user_id, event_type, event_ts (timestamp), page
events = spark.table("clickstream.events")

SESSION_GAP_SECONDS = 1800  # 30 min inactivity = new session

user_window = Window.partitionBy("user_id").orderBy("event_ts")

sessions = events \
    .withColumn("prev_ts", F.lag("event_ts").over(user_window)) \
    .withColumn("gap_seconds", 
        F.unix_timestamp("event_ts") - F.unix_timestamp("prev_ts")
    ) \
    .withColumn("is_new_session",
        (F.col("prev_ts").isNull() | (F.col("gap_seconds") > SESSION_GAP_SECONDS)).cast("int")
    ) \
    .withColumn("session_id",
        F.concat(
            F.col("user_id"),
            F.lit("_"),
            F.sum("is_new_session").over(user_window)
        )
    )

# Compute session-level aggregates
session_summary = sessions.groupBy("user_id", "session_id").agg(
    F.min("event_ts").alias("session_start"),
    F.max("event_ts").alias("session_end"),
    F.count("*").alias("event_count"),
    F.collect_list("page").alias("page_sequence"),
    (F.unix_timestamp(F.max("event_ts")) - 
     F.unix_timestamp(F.min("event_ts"))).alias("duration_seconds")
).withColumn(
    "converted", F.array_contains(F.col("page_sequence"), "checkout_complete")
)

# Edge case: sessions with single event have duration=0 (not null)
# Edge case: user_id=null events (anonymous) — keep but session_id will be "null_1"

SQL — Consecutive Date Gap Analysis (Hard)

sql · advanced · consecutive active days + longest streak
-- Find longest streak of consecutive active days per user
-- Handle: same user multiple events per day (dedup first)
-- Handle: gaps in dates (not just row differences)
WITH daily_activity AS (
    SELECT DISTINCT user_id, DATE(event_ts) AS activity_date
    FROM events
),
numbered AS (
    SELECT
        user_id, activity_date,
        ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY activity_date) AS rn
    FROM daily_activity
),
groups AS (
    SELECT
        user_id, activity_date,
        -- If consecutive, date - rn is constant → same group
        DATE_SUB(activity_date, rn) AS grp
    FROM numbered
),
streak_lengths AS (
    SELECT
        user_id, grp,
        MIN(activity_date) AS streak_start,
        MAX(activity_date) AS streak_end,
        COUNT(*) AS streak_days
    FROM groups
    GROUP BY user_id, grp
)
SELECT
    user_id,
    MAX(streak_days) AS longest_streak,
    FIRST_VALUE(streak_start) OVER (
        PARTITION BY user_id
        ORDER BY streak_days DESC
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS best_streak_start
FROM streak_lengths
GROUP BY user_id, streak_start  -- suboptimal: rewrite with QUALIFY or subquery
ORDER BY longest_streak DESC;

PySpark — Vectorized Pandas UDF (Production Pattern)

pyspark · pandas udf · feature engineering at scale
import pandas as pd
from pyspark.sql.functions import pandas_udf
from pyspark.sql.types import DoubleType

# Vectorized UDF: compute revenue percentile within segment
# Much faster than row-wise Python UDF

@pandas_udf(DoubleType())
def normalize_revenue(revenue: pd.Series, segment_mean: pd.Series, segment_std: pd.Series) -> pd.Series:
    """Z-score normalization within segment"""
    # Avoid division by zero for segments with constant revenue
    std_safe = segment_std.where(segment_std > 0, other=1.0)
    return (revenue - segment_mean) / std_safe

# Compute segment stats first (cheap aggregation)
segment_stats = df.groupBy("segment").agg(
    F.mean("revenue").alias("seg_mean"),
    F.stddev("revenue").alias("seg_std")
)

result = df.join(segment_stats, on="segment", how="left") \
    .withColumn("revenue_zscore", 
        normalize_revenue(F.col("revenue"), F.col("seg_mean"), F.col("seg_std"))
    )

# Edge case: null revenue → zscore = null (handled by pandas naturally)
# Edge case: single-member segment → std=0 → protected by std_safe above

⚠️ Trick Questions & Common Mistakes ▼
Trick: "df.collect() inside a UDF — what happens?"

Catastrophic. collect() is an action that pulls all data to the driver from within an executor task. This causes a nested Spark context issue and usually hangs or OOMs. Never call Spark actions inside UDFs.

Trick: ROWS BETWEEN vs RANGE BETWEEN with duplicates

If 3 rows have the same ORDER BY value, RANGE treats them as one unit (all get included). ROWS counts them individually. SUM with RANGE on duplicate timestamps can give inflated results.

D4

Data Streaming — Kafka + Spark Structured Streaming

Watermarking · Late Data · Exactly-Once · State · Kafka Internals · Failures

🧠 Concepts to Learn (Deep) ▼

1. Kafka Internals

  • Topic → Partitions → Offsets: Messages in a partition are ordered and immutable. Offset is the position. Consumers track offset independently.
  • Consumer Groups: Each partition assigned to exactly one consumer in a group. More consumers than partitions = idle consumers. Rebalancing triggered on join/leave/failure.
  • Retention vs Compaction: Time-based retention (delete old) vs Log compaction (keep latest value per key). Compacted topics ideal for CDC/changelogs.
  • Producer ACKs: acks=0 (fire-forget), acks=1 (leader ack), acks=all (all ISR ack). acks=all + min.insync.replicas=2 → strong durability guarantee.
  • Exactly-once semantics (EOS) in Kafka: Idempotent producer (dedup retries) + transactional API (atomic write across partitions). enable.idempotence=true + transactional.id.

2. Spark Structured Streaming

  • Micro-batch vs Continuous Processing: Micro-batch (default) processes data in intervals. Trigger.Once for scheduled batch-like runs. Trigger.Continuous for millisecond latency (experimental).
  • Checkpointing: Stores offset + state + query metadata to HDFS/ADLS. Required for exactly-once. Location: .option("checkpointLocation", path). Must be unique per query.
  • Output Modes: Append (new rows only), Complete (full result table), Update (changed rows). Aggregations require Complete or Update. Append + aggregation = only works with watermark.

3. Watermarking & Late Data

  • Watermark definition: withWatermark("event_time", "10 minutes") tells Spark: discard any event with event_time < max(event_time) - 10 min. This bounds state size.
  • Late data handling: Events within watermark window → included in aggregation. Events beyond watermark → dropped (no way to include without unbounded state).
  • Multiple watermarks (stream-stream join): Spark uses min watermark of both streams. One slow stream blocks state cleanup of the other.
  • Global watermark vs per-partition: Spark tracks watermark globally across partitions. A single stalled partition delays the global watermark.

4. Exactly-Once End-to-End

  • Source: Kafka with Spark offset tracking (committed only after successful write).
  • Processing: Spark uses idempotent writes + transactional commits per micro-batch epoch.
  • Sink: Delta Lake (transactional) or Kafka (transactional API). File sinks with foreachBatch + atomic moves.
  • foreachBatch pattern: Most flexible — gives you a DataFrame per batch. Use for complex sinks. Must handle idempotency yourself (write to temp + atomic rename, or use Delta MERGE).

5. Stateful Operations

  • mapGroupsWithState: Custom state management. Arbitrary state per key. You control state schema, update, and timeout.
  • flatMapGroupsWithState: Like above but can emit 0 or more rows per state update.
  • State store: In-memory + checkpointed to HDFS. RocksDB state store (Databricks) for large states — off-heap, spills to disk.

from pyspark.sql import functions as F
from pyspark.sql.types import StructType, StructField, StringType, LongType, TimestampType

EVENT_SCHEMA = StructType([
    StructField("event_id", StringType()),
    StructField("user_id", StringType()),
    StructField("event_type", StringType()),
    StructField("amount", LongType()),
    StructField("event_time", TimestampType()),
])

# Read from Kafka with explicit offset management
raw_stream = spark.readStream \
    .format("kafka") \
    .option("kafka.bootstrap.servers", "broker1:9092,broker2:9092") \
    .option("subscribe", "user_events") \
    .option("startingOffsets", "latest") \
    .option("maxOffsetsPerTrigger", 100_000)  # backpressure \
    .option("failOnDataLoss", "false")  # handle Kafka retention \
    .load()

# Parse JSON payload
parsed = raw_stream \
    .select(
        F.from_json(F.col("value").cast("string"), EVENT_SCHEMA).alias("data"),
        F.col("offset"), F.col("partition"), F.col("timestamp").alias("kafka_ts")
    ) \
    .select("data.*", "offset", "partition", "kafka_ts") \
    .filter(F.col("event_id").isNotNull())  # drop malformed

# Watermark: allow up to 10 min late arrival
# Events older than (max_event_time - 10min) are DROPPED
windowed = parsed \
    .withWatermark("event_time", "10 minutes") \
    .groupBy(
        F.window("event_time", "5 minutes"),  # 5-min tumbling window
        "event_type"
    ).agg(
        F.count("*").alias("event_count"),
        F.sum("amount").alias("total_amount"),
        F.countDistinct("user_id").alias("unique_users")
    )

# Write with exactly-once via foreachBatch + Delta MERGE
def upsert_to_delta(batch_df, batch_id):
    from delta.tables import DeltaTable
    
    if batch_df.isEmpty():
        return
    
    # Deduplicate within batch (Kafka may deliver duplicates on retry)
    deduped = batch_df.dropDuplicates(["window", "event_type"])
    
    target = DeltaTable.forPath(spark, "/mnt/gold/event_aggregates")
    target.alias("tgt").merge(
        deduped.alias("src"),
        "tgt.window = src.window AND tgt.event_type = src.event_type"
    ).whenMatchedUpdateAll() \
     .whenNotMatchedInsertAll() \
     .execute()

query = windowed.writeStream \
    .foreachBatch(upsert_to_delta) \
    .option("checkpointLocation", "/mnt/checkpoints/event_aggregates") \
    .trigger(processingTime="1 minute") \
    .start()

query.awaitTermination()

⚠️ Trick Questions & Edge Cases ▼
Trick: "Can you use aggregations with Append output mode?"

Only if a watermark is defined. Without watermark, Spark doesn't know when to finalize a window, so Append mode is rejected. With watermark, Spark emits window results only after the watermark passes the window end.

Edge Case: Checkpoint schema change

If you change the streaming query (add/remove columns, change aggregation), the checkpoint may be incompatible. You must delete the checkpoint and restart — losing offset position. Design: always make checkpoint path versioned (v1, v2...) for safe schema migrations.

Trick: failOnDataLoss=false — what does it mean?

If Kafka deletes messages (retention expired) before Spark reads them, with failOnDataLoss=true the query fails. With false it silently skips. You MUST have a DLQ or monitoring to detect this data loss.

Data Modeling + Orchestration + CI/CD

Star Schema · SCD · Medallion Architecture · Airflow · Testing · Observability

🧠 Concepts to Learn (Deep) ▼

1. Data Modeling for the Lakehouse

  • Star Schema: Fact table (measurements/events) surrounded by denormalized dimension tables. Fast reads, easy BI queries. No joins between dims.
  • Snowflake Schema: Normalized dimensions (dims have their own dims). Less storage, more joins. Use only when dim cardinality is extreme.
  • Medallion Architecture (Bronze/Silver/Gold): Bronze = raw ingestion (schema-on-read, no transformation). Silver = cleansed, validated, deduped. Gold = business-level aggregates, domain models. Know when to skip Silver for streaming paths.
  • One Big Table (OBT): Pre-joined, denormalized, optimized for specific query pattern. Trade-off: write amplification, update complexity vs read performance.
  • Data Vault: Hub (business keys), Link (relationships), Satellite (attributes). Highly scalable, audit-friendly, but complex for BI. Know when to recommend vs not.

2. Airflow Deep Dive

  • Concepts: DAG, Task, Operator, XCom, Connections, Variables, Sensors, TaskGroups.
  • Scheduler internals: DAG parsing loop, task queuing, executor model (LocalExecutor, CeleryExecutor, KubernetesExecutor). Know differences.
  • Dynamic DAGs: Generate DAGs programmatically from config/DB. Use with caution — DAG parsing delay, serialization issues.
  • Backfill strategy: airflow dags backfill --start-date --end-date. Idempotency requirement: each DAG run must be re-runnable without duplicates.
  • Sensors vs Deferrable Operators: Sensors hold a worker slot while waiting (expensive). Deferrable operators release worker, use async triggerer. Use deferrable in production.
  • XCom limits: XCom stored in metadata DB. Not for large data. Max ~48KB in Postgres. Use S3/ADLS paths instead.

3. CI/CD for Data Pipelines

  • Testing pyramid: Unit tests (pure functions, transformations) → Integration tests (against test Delta tables) → Contract tests (schema validation) → E2E tests (full pipeline on sample data).
  • dbt testing pattern: not_null, unique, accepted_values, relationship tests. Custom tests with SQL assertions.
  • Great Expectations: Expectation Suites, Checkpoints, Data Docs. Integrate in pipeline as validation step before promotion to Gold.
  • Schema registry: Avro/Protobuf with Confluent Schema Registry for Kafka. Prevents breaking schema changes from reaching consumers.

4. Observability

  • Data quality metrics: completeness (null rate), uniqueness (dedup rate), freshness (max event_time lag), volumetric (row count vs expected), distribution drift.
  • Pipeline SLAs: Alert if Gold table not refreshed by 6am. Alert if Silver row count drops >20% vs 7-day average.
  • Spark metrics: Prometheus + Grafana on Databricks. Track: job duration, shuffle bytes, GC time, failed tasks, executor OOM count.

Coding Problem · Hard · 45 min
You have a transaction table: (txn_id, account_id, amount, txn_type, txn_timestamp, status). Write PySpark code to:

1. Flag accounts where 3 or more transactions failed within any 1-hour window
2. For each flagged account, compute: total attempted amount, success rate, first failure timestamp
3. Join with accounts table to get account holder name and tier
4. Write result to Delta with partitioning by account_tier and date
5. Handle: null amounts, duplicate txn_ids, accounts with no failures
↳ Follow-up: "Your result has 50,000 small files after 30 days. How do you fix this?"
↳ Follow-up: "Add a streaming version that detects this in real-time with 5-minute trigger."

expected solution skeleton
from pyspark.sql import functions as F, Window

txns = spark.table("finance.transactions") \
    .dropDuplicates(["txn_id"]) \
    .withColumn("amount", F.coalesce(F.col("amount"), F.lit(0.0)))

# Sliding window: count failures in 1-hour window before each txn
w = Window.partitionBy("account_id") \
    .orderBy(F.unix_timestamp("txn_timestamp")) \
    .rangeBetween(-3600, 0)

flagged_accounts = txns \
    .withColumn("is_failed", (F.col("status") == "FAILED").cast("int")) \
    .withColumn("failures_in_window", F.sum("is_failed").over(w)) \
    .filter(F.col("failures_in_window") >= 3) \
    .select("account_id").distinct()

account_stats = txns.join(flagged_accounts, on="account_id", how="inner") \
    .groupBy("account_id").agg(
        F.sum("amount").alias("total_attempted"),
        (F.sum((F.col("status") == "SUCCESS").cast("int")) /
         F.count("*")).alias("success_rate"),
        F.min(F.when(F.col("status") == "FAILED", F.col("txn_timestamp")))
         .alias("first_failure_ts")
    )

accounts = spark.table("dim.accounts")
result = account_stats.join(accounts, on="account_id", how="left") \
    .withColumn("report_date", F.current_date())

result.write.format("delta") \
    .partitionBy("account_tier", "report_date") \
    .mode("overwrite") \
    .option("partitionOverwriteMode", "dynamic") \
    .save("/mnt/gold/flagged_accounts")
